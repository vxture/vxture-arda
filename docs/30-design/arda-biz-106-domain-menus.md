## 0. 设计原则 (owner 评审后修订)

1. **菜单数量由业务覆盖决定**, 不设条数上限; 参照行业产品 (华为 DataArts Studio,
   阿里 DataWorks/Dataphin, Collibra/Alation/Atlan/Purview/DataHub, 亿信睿治) 的
   同域功能划分, 需要多少建多少。
2. **场景动线决定相邻性**: 菜单按真实工作者的日常动线组织 (biz-110 SS4 角色表),
   而非按概念分类。质量-血缘(元数据)-问题管理(治理) 是 steward 的高频铁三角。
3. **一条数据一个家**: 审批唯一承载在管理域"审批中心"; 各域出现的是深链, 不建
   重复审批页。跨域动作 (服务密钥/主数据服务等) 一律页内深链, 不做跨板块菜单项。
4. **占位即菜单**: 未建功能直接进菜单 (`NavItem.future`), 路由到统一的
   `UnderConstruction` 占位页; `biz-105` round-1 的域级 `DOMAIN_ROADMAP` chips 退役
   —— 路线图现在就是菜单本身。
5. 门控照旧: `SCREEN_FEATURES` 按屏登记, 子屏幕沿用所在域已有能力键 (先键后域,
   新键须先入 `biz-120` SS3.1)。launcher 的分组/归集方式**本文不定**, 菜单立住后另行设计。

## 3. 骨架落地要点 (round 2)

- `nav-config.ts`: BOARD_NAV 全量重写为上表; BOARDS.home 指向各域首个可达项,
  header launcher 经 `ROUTE_BY_KEY` 解析 home 路由。
- 路由: 子屏幕嵌套在域锚点目录下 (`/quality/results`, `/standards/bindings`...),
  自动继承域 layout 的 `ScreenGate` 门控; 新增无锚点域目录 (`/metadata/*`,
  `/integration/*`, `/approvals`, `/admin/*`, `/todo`) 暂不设 layout 门控 (纯占位,
  无数据面; 真实建设时按 SS0.5 登记门控)。
- `shell.tsx` activeKey 从"路径首段"改为 **NAV_FLAT 最长路由前缀匹配**
  (支持嵌套路由与详情页归属)。
- 占位组件: `UnderConstruction` (screenKey + icon -> 页标题 + 建设中空态);
  `DomainRoadmap`/`DOMAIN_ROADMAP`/i18n `domainRoadmap.*` 退役删除。
  规划/架构/治理域根路由重定向到首个菜单项。
- `SCREEN_FEATURES`: 子屏幕按 SS0.5 映射到既有键 (标准子屏->`governance.standards`,
  质量子屏->`governance.quality_rules`, 安全分级子屏->`governance.classification`,
  策略子屏->`governance.policies`, 运营子屏->`operations.dashboard` 等);
  无既有键的子屏 (集成同步/健康度, 元数据采集/变更/质量, 待办/审批/字典/通知)
  暂不登记, 待键目录补录。
