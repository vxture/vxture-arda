## 0. 产品主张：为什么这样分域能打动用户

用户不关心"我们有 13 个能力维度"，只关心**"我的数据到底怎么样、能不能信、怎么用"**。两轴分域就是把这份关心翻译成 5 个"一站式主场"：

- **让每类用户在一个域里把事办完**（减少页面跳转）——域即角色的主场（§4）。
- **让"可信"一眼可见**——数据的质量/血缘/分级/主数据/标准符合性，**汇聚到一张数据画像页**，不用东拼西凑（§2.1 的打动点）。
- **让治理成效立刻兑现**——在治理域做一个动作，资产域的画像、服务域的输出**当场变化**（§3 跨域价值闭环）。

> 一句话：**资产域让用户"信"，治理域让数据"值得信"，集成域"接进来"，服务域"用出去"，管理域"管得住"。**

## 1. 两轴速览（承 `biz-100` §1.2）

**矩阵**：**纵 = 能力域**（一个能力贯穿其生命周期 目标→过程→结果→服务→监管）× **横 = 应用视角/阶段**（固定一个阶段拉通所有能力 = 功能域）。两种导航都要支持：

- **横向（按域/阶段，主 IA）**：一个角色在一个域内拉通所有能力（下表 5 域，即生命周期的结果/入/过程/服务/监管阶段）。
- **纵向（按能力，能力专题）**：一个能力负责人端到端看一条能力线（如质量经理看质量的目标→过程→结果→服务→监管）。作二级导航/过滤（§2.6）。

| 域（= 阶段） | 一句话 | 用户问的问题 |
|---|---|---|
| 资产域 · 看（结果） | 有什么数据、可不可信 | "我要的数据在哪？能信吗？多少量？" |
| 集成域 · 入（接入执行） | 把数据接进来、保鲜 | "怎么把这个源接进来？还新鲜吗？" |
| 治理域 · 管（目标+过程） | 让数据从"有"变"可信" | "怎么定标准/查质量/做脱敏/治主数据？" |
| 服务域 · 出（服务） | 把可信数据变成可用服务 | "怎么把它发成 API？用得怎么样？" |
| 管理域 · 运营（监管） | 平台的账本与开关 | "谁动了数据？密钥/留存/容量怎么管？" |

> 完整 **能力 × 阶段矩阵**见 `biz-100` §1.2。本文按**横向 5 域**详解（§2.1-2.5），并在 §2.6 给出**纵向能力专题**导航。

## 2. 五域详解

每域统一讲：**主用户 → 用户意图 → 打动点(wow) → 看什么(成果面) / 做什么(过程面) → 关键屏幕 → 覆盖能力 → 门控要点**。

### 2.1 资产域（看：有什么 / 有多少 / 什么状态）

- **主用户**：数据消费者、数据 owner、业务分析师、外部消费方/智能体（经服务域取，但在此"发现+判断可信"）。
- **用户意图**：找到数据、判断能不能用、看懂它的全貌与可信度。
- **打动点（wow）**：**一张"数据画像页"看全一份数据的家底与可信度**——打开任一数据集，一页同时看到：口径/负责人/更新频率/量级、字段结构、**质量分与趋势**、**血缘上下游**、**安全分级/敏感标识**、**是否主数据(金记录)**、**符合哪些标准**、标签/术语。用户不再在 5 个菜单间拼图，"**可信一眼可见**"。
- **看什么（成果面，只读汇聚——把治理成果搬到这里展示）**：
  - **发现**：目录（列表/卡片）、域/团队/标签筛选、搜索（含高级/语义搜索）。
  - **数据画像（详情页）**：基本信息 + 元数据/字段 + 质量分 + 血缘图 + 安全分级 + 主数据标识 + 标准符合性 + 术语。
  - **"有多少"**：资产总量、按域/团队/分级分布、存储占用（容量画像）。
- **做什么（域内轻动作）**：申请访问、收藏、（有权则）编辑元数据/打标签、**一键跳"去治理"**（转治理域对应工作台）。
- **关键屏幕**：`catalog`（发现）、`catalog/[id]` 数据画像（**重点重排页**：纳入质量/血缘/安全/主数据/标准的只读面）。
- **覆盖能力**：资产、元数据；**成果面**的标准/主数据/质量/血缘/安全；存储（有多少）。
- **门控**：看 = baseline（`arda.assets.catalog`）；高级搜索/编辑元数据 = 对应键 + 相应角色。

### 2.2 集成域（入：接入 / 更新）

- **主用户**：数据工程师、集成管理员。
- **用户意图**：接一个新数据源、让数据持续保鲜。
- **打动点（wow）**：**几步接入 → 自动编目 → 立刻出现在资产画像**——登记源、拉元数据后，资产域立刻多出可发现、带画像的资产；**更新健康一屏可见**（谁新鲜、谁滞后、谁失败）。
- **看什么（成果面）**：数据源列表、连接状态、最近同步时间、**更新健康度/新鲜度**。
- **做什么（过程面）**：登记数据源（类型/连接配置，应用层加密）、连接测试、拉取元数据、（future）配管道/调度/实时。
- **关键屏幕**：数据源登记/管理（已建，`sources`）、`etl`（已建，管道运行/调度/监控，非占位）。
- **覆盖能力**：集成/更新。
- **门控**：`sources_basic`/`sources_premium`；`pipelines`/`scheduling`/`realtime` = future 占位（区别于"升级"文案）。
- **导航现状（`biz-105`）**：`sources` 挂在导航域"数据集成"；`etl` 拆到独立的导航域"数据开发"（对应用户能力地图里的"Engineering"），本轮未新增门控键、维持未门控。两者仍同属本节"集成"能力语义，只是导航呈现拆成了两个入口。

### 2.3 治理域（管：怎么治）—— 治理工作台

- **主用户**：数据 steward、治理员、安全/合规负责人、数据 owner（治理侧）。
- **用户意图**：给数据定标准、查质量、做脱敏、管血缘、治主数据、定留存。
- **打动点（wow）**：**治理动作即时兑现为可见成效**——建一条质检规则→跑批→**资产画像上的质量分当场变**；定一条脱敏策略→**对外服务当场生效**。治理不再是"填表归档"，而是**看得见结果的工作台**（这是与普通数据目录的根本差异：治理即信任，§3）。
- **做什么（过程面/工作台）**：
  - **标准化**：建/发布/评审数据标准（代码集/数据元）。
  - **质检**：建质量规则、跑检查、看六维评估、处理告警。
  - **安全**：配分级规则、脱敏/加密策略、访问策略、共享审批。
  - **血缘**：打标/校正血缘、影响分析（改这份数据会影响谁）。
  - **主数据**：标金记录、指派 steward、（轻量）匹配/合并；重型 MDM 引擎深度待定（`biz-100` §1）。
  - **生命周期**：定留存规则（执行在管理域）。
- **看什么**：治理工作台的规则/策略/结果列表、告警、覆盖率（治理覆盖了多少资产）。
- **关键屏幕**：`standards`、`quality`、`security`、`lineage`（**接库重排**）；主数据治理入口（新增，`masterdata`，已建占位）。
- **覆盖能力**：标准/主数据/质量/血缘/安全/策略/生命周期（**过程面**）。
- **门控**：各能力键（`arda.governance.*`）；**写操作需更高角色**（steward/admin），查看可放宽。
- **导航现状（`biz-105`）**：本节列的能力不再挤在一个"治理域"导航里，各自有独立导航域——`standards`→"数据标准"、`quality`→"数据质量"、`security`→"数据安全"、`lineage`→"元数据"（血缘/影响分析）、主数据入口→"主数据"。本节"做什么"列出的**治理组织/责任人/审批流程/制度/流程/问题管理/整改/治理评价**这些组织与流程类能力，在新导航里归入独立的"数据治理"占位域（`arda.governance.workbench`，future），与上述已上线的管控键并存，不是同一回事——门控能力键的语义与归属不变，只是导航呈现更细。

### 2.4 服务域（出：数据服务 / 共享 / 监测）

- **主用户**：数据服务开发者、数据产品经理、外部对接人。
- **用户意图**：把可信数据封装成 API/查询/导出、对外共享、看用得怎么样。
- **打动点（wow）**：**从可信资产一键发布为服务 + 调用可观测**——选一份（画像已达标的）资产→发布为 API/数据产品→拿到**调用统计与监测**；**对外契约（隔离/分级过滤/审计/配额）自动收口**，开发者不用自己重造安全。
- **看什么（成果面）**：服务/API 目录（**注意：这是"服务目录"，不同于资产域的"数据目录"**）、发布状态、调用统计、**服务监测**（健康/用量/错误率）。
- **做什么（过程面）**：发布/编辑/下线服务、配置对外共享、关联/管理调用密钥（密钥实体在管理域）。
- **关键屏幕**：`service`（目录+发布+监测）。
- **覆盖能力**：数据服务/共享、服务监测。
- **门控**：`arda.services.publish_api`/`data_products`；发布/下线 = 更高角色；对外请求量受 `api_requests_monthly` 配额。

### 2.5 管理域（运营：审计 / 生命周期 / 密钥 / 容量）

- **主用户**：平台管理员、安全/合规、owner。
- **用户意图**：看谁动了数据、管凭证、控留存/销毁、看容量。
- **打动点（wow）**：**全链路可审 + 一处运维**——"谁在何时经哪个服务取了哪份数据"一目了然（合规底气）；密钥、留存、容量在一处管住。
- **看什么/做什么**：审计日志（检索/过滤/导出）、API Key 管理（创建一次性明文、吊销）、**生命周期执行**（到期归档/销毁、平台 `wipe` 软删+延迟硬删）、存储容量运营/配额告警。
- **关键屏幕**：API Key 管理（`apikeys`，已建）、审计日志（`audit`，已建）；生命周期执行/存储容量运营界面仍待建。
- **覆盖能力**：审计、生命周期（执行）、密钥、存储（运营）。
- **门控**：**权限为主**（owner/admin）；配额为辅。
- **导航现状与平台边界（`biz-105` §2）**：用户能力地图里的"系统管理"若字面包含组织/租户/用户/角色/菜单/字典/参数/License/插件，其中组织/租户/用户/角色/License **属于 vxture 平台（accounts.vxture.com / OIDC），不是 arda 的职责**（见 `biz-100` §0）。本域导航范围维持仓库既有边界——只有 API Key 与审计日志——不在 arda 内假建一套重复的身份管理界面；如需入口，建议做成跳转至 vxture 平台管理控制台的深链接。

### 2.6 纵向：能力专题（按能力端到端，第二导航）

§2.1-2.5 是**横向**（按域/阶段）。产品还要支持**纵向**——固定一个**能力域**，端到端看它的 目标→过程→结果→服务→监管，跨域拉成一条线。这是给**能力负责人**的主场：

| 能力专题 | 谁用 | 端到端一条线（目标→过程→结果→服务→监管） | 打动点 |
|---|---|---|---|
| **质量专题** | 质量经理 | 定 SLA → 建规则/跑质检 → 质量分/报告 → 质量准入 → 合规审计 | 一个页面掌控全域质量健康，不用在治理/资产/管理间跳 |
| **安全专题** | 安全/合规官 | 定分级/合规 → 脱敏/加密/授权 → 分级分布 → 安全过滤服务 → 访问审计 | 数据安全端到端一屏可控可审 |
| **主数据专题** | 主数据 steward | 定模型/权威源 → 匹配/合并 → 金记录 → 主数据服务 → 变更审计 | 主数据全生命周期集中治理 |
| **标准专题** | 标准管理员 | 定标准 → 落标/评审 → 符合性 → 标准约束 → 合规审计 | 标准从制定到落地到审计闭环 |
| **血缘专题** | 数据架构师 | 覆盖目标 → 采集/校正 → 血缘图/影响 → 可溯源服务 → 审计 | 影响分析 + 溯源一站式 |

> 实现上：能力专题是**跨域的一条过滤线**（同一批能力键 + 该能力相关实体），不是新增数据；是把横向 5 域里属于"该能力"的格子纵向串起来（对应 `biz-100` §1.2 矩阵的一整行）。作二级导航或"专题工作台"。

## 3. 跨域价值闭环（产品主线，最能打动用户）

两轴分域的最大价值不在"分得清"，而在**四个域连成一条用户看得见的闭环**：

```
   集成域(入)                治理域(管)                资产域(看)              服务域(出)
   接入数据源  ──自动编目──▶  建质检/定分级/治主数据 ──成效即时回写──▶ 数据画像可信度↑ ──一键发布──▶ 可信数据服务
       │                          │                                             │
       └──────────────────────────┴────────── 全程可审 / 可溯源 (管理域·运营) ───┘
```

- **接进来的数据**（集成）→ **被治理成可信**（治理，成效即时回写资产画像）→ **在画像里"可信一眼可见"**（资产）→ **一键变成可用服务**（服务）→ **全程可审可溯源**（管理）。
- 这条闭环是**"治理即信任"落到用户体验**的证据：用户能亲眼看到"我做了治理，数据就更可信、更好用了"。**普通数据目录只做资产域(看)一半；arda 的差异化是把闭环跑通。**

## 4. 角色 → 主域（减少页面跳转）

下表的"主场域"是**能力语义**（本文 §2 的 5 域）；括号内标出 2026-07-15 后对应的
**导航域**名称（`biz-105`），供对照控制台实际菜单。

| 角色 | 主场域（能力） | 对应导航域 | 偶尔去 |
|---|---|---|---|
| 业务/消费用户、分析师 | 资产域（看） | 数据资产 | 数据服务（取数） |
| 数据 steward / 治理员 | 治理域（工作台） | 数据标准 + 数据质量 + 数据安全 + 主数据 + 数据治理（组织/流程） | 数据资产（看成效） |
| 安全 / 合规 | 治理域（安全）+ 管理域（审计） | 数据安全 + 管理（审计） | 数据资产（分级分布） |
| 数据工程师 | 集成域（接数） | 数据集成 + 数据开发（跑管道） | 数据资产（确认编目） |
| 数据服务开发/产品 | 服务域（发布/监测） | 数据服务 | 数据资产（选资产） |
| 平台管理员 / owner | 管理域（运营） | 管理 | 全域 |

> 设计目标：**一个角色的高频任务尽量收敛在其主场域内**，跨域是"顺流而下"（接入→治理→发布），不是来回横跳。导航域拆得更细（尤其治理域拆成 5 个）不改变这个目标——同一角色的高频任务现在分布在几个相邻的导航域里，而不是一个大而全的"治理工作台"页面。

## 5. 对板块详细设计（biz-2xx）的重排影响

采纳两轴后，`biz-210..250` 的技术详细设计范围按下表对齐（**本文定模型，`biz-2xx` 逐篇重排另行**）：

| 板块 | 重排要点 |
|---|---|
| `biz-210` 资产域 | **重点**：数据画像页纳入质量分/血缘/安全分级/主数据标识/标准符合性的**只读成果面**（数据来自治理域实体，此处只展示）；补"有多少/容量画像" |
| `biz-220` 集成域 | 补更新健康/新鲜度视图；数据源登记界面（已列待办） |
| `biz-230` 治理域 | 收敛为**过程面/工作台**（标准/质量/安全/血缘/主数据/策略/生命周期规则的"做"）；成果展示移交资产域 |
| `biz-240` 服务域 | 明确"服务目录 ≠ 数据目录"；补服务监测；对外契约不变量（已有） |
| `biz-250` 管理域 | 补生命周期执行、存储容量运营；审计/密钥界面（已列待办） |
| **纵向·能力专题** | **新增二级导航**（§2.6）：质量/安全/主数据/标准/血缘各一条端到端线，跨域拉通同一能力的目标→过程→结果→服务→监管——不新增数据，是矩阵一整行的聚合视图 |

> 门控不变（`biz-100` §1.2）：同一能力在不同阶段门控轻重不同（结果/看 轻、过程/管 重、监管属管理域）；重排只动 IA/屏幕归属，不动能力键定义。
