## 0. 两层区分：看和组织 vs 功能

| 层 | 是什么 | 归谁 / 何时 |
|---|---|---|
| **看和组织** | 功能怎么切换、页面怎么归组、导航 | header **九宫 launcher**，后续逐步落地（`biz-110`/`biz-2xx`） |
| **功能**（本系列 `biz-4xx`） | 每个功能**端到端能不能用** | **现在设计**，与"在哪显示"无关 |

**为什么按功能设计、不按导航**：导航解决"看得到"，功能解决"用得了"。一个功能哪怕导航做得再漂亮，若"目标定了没执行器""结果出了服务用不上""改了没审计"——就是**断链 = 没贯通 = 没真正实现**。本系列**逐功能沿生命周期查断链、给实现**。

## 2. 功能设计模板（每个 `biz-4xx` 统一结构）

1. **功能定义**：一句话 + 它保证什么。
2. **贯通链（目标 → 过程 → 结果 → 服务 → 监管）**：逐环写「做什么 · 靠什么实现（实体/操作）· 怎么接下一环」。
3. **断链清单**：当前哪些环没接通 / 没实现（编号 `<X>-BL<n>`，喂 §3 与 `biz-300`）。
4. **数据模型（实现）**：实体 + 字段 delta（以 arch 维度为准）+ 关键操作。
5. **依赖**：其他功能 / 平台服务。
6. **门控**：能力键（`arda.*`，**不涉及导航**）。

> **贯通判据**：五环每一环都有实现、且**上一环产出正好是下一环输入**（目标产规则→过程消费规则产结果→结果供服务与监管）。任一环缺实现或接口错位 = 断链。

## 3. 断链总看板（跨功能 · 实施直接输入）

汇总各功能的断链；这张表**就是 `biz-300` 实施计划的 backlog**。逐条见各 `biz-4xx` §3。

**各功能断链汇总**（关键断链 = 该功能"不通"的卡点）：

| 功能 | 断链编号 | 关键断链（高优先） |
|---|---|---|
| 数据接入 `biz-410` | I-BL1..4 | **I-BL1** 登记→拉元数据→生成资产执行链 |
| 数据资产 `biz-421` | A-BL1..3 | **A-BL1** 画像结果面聚合（依赖各功能） |
| 元数据 `biz-422` | MD-BL1..6 | **MD-BL1/BL3** 技术元数据采集 + 术语/标签策展 |
| 数据标准 `biz-431` | S-BL1..4 | S-BL1 符合性关联 |
| 主数据 `biz-432` | M-BL1..4 | M-BL1 金记录标记；M-BL2 重型引擎待定 |
| 数据质量 `biz-433` | Q-BL1..4 | **Q-BL1** 调度；**Q-BL2** 服务准入 |
| 数据血缘 `biz-434` | L-BL1..4 | **L-BL1** 血缘 UI 接库 |
| 数据安全 `biz-435` | Sec-BL1..5 | **Sec-BL1/BL2** 脱敏 + 对外分级过滤 |
| 数据服务 `biz-441` | Svc-BL1..5 | **Svc-BL1** 对外契约收口 |
| 生命周期 `biz-451` | Lc-BL1..4 | **Lc-BL3** 软删 schema 决策（前置） |

**关键路径（高优先 · 让功能"活起来"的最小集）**：
1. `I-BL1` 接入能生成资产（价值链起点，断了后面全空）。
2. `Q-BL1`+`Q-BL2` 质量能真跑 + 卡服务准入。
3. `Sec-BL1`+`Sec-BL2` 脱敏与对外分级过滤真生效。
4. `Svc-BL1` 对外契约收口（隔离已有，补分级/密钥/审计/配额）。
5. `L-BL1` 血缘接库。
6. `Lc-BL3` 软删 schema 决策（wipe/销毁的前置）。

**跨功能共性断链（一处补、多功能受益）**：
- **审计写入稀疏**：多功能的"监管"环都缺 `AuditLog` 写入（Q-BL3 / Sec-BL3 / Svc-BL5 / Lc-BL4 / …）→ 统一补写入封装。
- **服务出口收口**：质量准入 / 分级过滤 / 密钥 / 配额都在 `biz-441` 对外契约汇合。
- **调度缺失**：质量跑批（Q-BL1）、接入同步（I-BL2）、血缘采集（L-BL2）都缺 scheduling（future）。

> **断链 = 功能没贯通的点**；实施就是**把断链一条条接通**。`biz-300` 按上述关键路径 + 共性断链分阶段排期。

## 5. 编号说明（3 位数 · 按生命周期层归类）

- 功能层 `4xx`：**百位=功能层，十位=生命周期层，个位=层内序**（`biz-000` §0）：
  - `41x` 接入：`biz-410` 数据接入。
  - `42x` 编目：`biz-421` 数据资产、`biz-422` 元数据（**两个维度，各一篇**：资产=对象，元数据=描述）。
  - `43x` 治理：`biz-431` 数据标准、`biz-432` 主数据（**各一篇，不合并**）、`biz-433` 数据质量、`biz-434` 数据血缘、`biz-435` 数据安全。
  - `44x` 服务：`biz-441` 数据服务。
  - `45x` 运营：`biz-451` 数据生命周期。
- 用 3 位是为了**留足空间、按层归类、方便调整**——不再为编号不足而合并文档。
