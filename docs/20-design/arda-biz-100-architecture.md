## 0. 定位与边界

arda = vxture 生态里的 **`data` Product = 通用数据平台（数据域）**。它把企业数据**编目→治理→服务化**，产出「可信、受控、可溯源的数据」，供人（控制台用户）与外部消费方（含业务智能体、BI、下游数据产品）使用。

**一句话职责（分界线）**：arda 只管**数据本身**——**不管数据怎么用、在哪用**（用是消费方的事）。这条是 arda 与所有消费方（含智能体）的边界。

> 「管数据本身」的**完整能力维度见 §1**。常说的"资产 / 标准 / 治理 / 质量 / 服务 / 更新"是**面向用户的主干子集**，非全集；平台视角还须补上**元数据、血缘、安全、参考与主数据、生命周期、存储、审计**等维度（原则不变：仍只管数据本身，不管其使用）。

**只做数据域**：kb、业务智能体、RAG/向量、LLM、交付物生成属其他产品（见 `biz-000` §0）。arda 提供数据，不做智能体。

**数据类型范围**：聚焦**结构化数据**（库表 / 数仓 / 流）与**常规非结构化数据**（文档、图片、表格等以**文件形态**编目治理的数据资产）。**不涉及**地理信息（GIS）、三维、物联（IoT）等**复杂孪生数据**——那是另一产品的职责，arda 不建模、不处理这类空间/时序孪生数据。

> 关键区分：文档 / 图片在 arda 是**被编目治理的数据资产**（元数据 / 分级 / 质量 / 服务化）；对文档做切块 / 向量 / 检索 / 问答（RAG）是 kb / 智能体产品的事，不在 arda。同一份文档，arda 管"它是什么、可不可信、给谁用"，不管"用它回答问题"。

## 1. 业务能力全景（capability map · 平台视角补齐）

「管数据本身」的完整能力维度，按**数据管理生命周期**分五层（接入→编目→治理→服务→运营）。最初 6 项为主干，平台视角补上 6 项支撑维度：

| 层 | 能力维度 | 内涵 | 板块 | 相对初版 6 项 |
|---|---|---|---|---|
| 接入 | 集成 / 更新 | 数据源接入、元数据拉取、同步保鲜 | integration | 主干「更新」 |
| 编目 | 数据资产 | 登记 / 发现 / 目录 | assets | 主干「资产」 |
| 编目 | **元数据** | 技术 / 业务 / 操作元数据、schema、标签、术语、检索 | assets | **补齐**（资产的底座） |
| 治理 | 数据标准 / 参考数据 | 代码集 / 数据元 / 参考值（国家 / 币种码等） | governance | 主干「标准」 |
| 治理 | **主数据（MDM）** | 核心业务实体的权威金记录（客户 / 产品 / 供应商等）：金记录标注、steward、匹配 / 合并 | governance | **补齐（深度待定，见下）** |
| 治理 | 数据质量 | 规则 / 检查 / 六维评分 | governance | 主干「质量」 |
| 治理 | **数据血缘** | 来源 / 影响分析 | governance | **补齐**（升为一等维度） |
| 治理 | **数据安全** | 分级 / 脱敏 / 访问控制 | governance | **补齐**（升为一等维度） |
| 治理 | 治理策略 | 访问 / 脱敏 / 留存规则 | governance | 主干「治理」 |
| 治理 | **数据生命周期** | 留存 / 归档 / 销毁（wipe） | governance + admin | **补齐**（原 6 项完全没有） |
| 服务 | 数据服务 / 共享 | API / 查询 / 导出 / 对外共享 | services | 主干「服务」 |
| 运营 | **存储** | 存储 / 容量管理（元数据 + 文件资产对象存储） | 跨切面 | **补齐**（文档 / 图片资产尤需） |
| 运营 | **审计** | 活动 / 平台指令审计、幂等防重放 | admin | **补齐**（支撑面） |

> 「能力维度」= arda 管什么（13 项）；「板块」= 门控 / IA 的组织单位（5 个，`overview` 恒开除外）。多维度可归同一板块（governance 承载标准 / 参考主数据 / 质量 / 血缘 / 安全 / 策略 / 生命周期）。
> 补齐维度的实现成熟度不一：血缘 / 安全 / 元数据已有实体或屏幕（见 `biz-210`/`biz-230`）；**生命周期（留存 / 销毁）、存储、主数据目前薄或空**（`arda.quota.history_retention_days`/`storage_bytes` 已有键，wipe 见 ADR §5.1 未实现；主数据见下）——落地见 `biz-300`。
> **主数据（MDM）深度决策**：arda 天然承担主数据的**编目 / 治理 / 标准 / 质量 / 服务化**（把主数据资产标为权威金记录、指定 steward、加质量规则、发布主数据服务）——与现有 `Dataset`+steward+`QualityRule` 模型一致，轻量即可落地。**重型 MDM 引擎**（实体匹配 / 合并 / 存活规则 survivorship / 建模工作流）是否在 arda 内建为**待定深度项**：可作 governance 增值能力，或独立成 MDM 能力/产品（类比 GIS/孪生"另有产品"，但主数据更核心，倾向留在 arda）。**参考数据**（代码集 / 数据元）已由 `Standard` 覆盖。

### 1.1 功能域如何划分（域 vs 维度）

能力**维度**（13）多于功能**域**（5），故按"**同一门控 / 运营单位**"归并，而非一维一域——一维一域会让门控矩阵爆炸（functional-domains §2.2 已论证）。5 域：`assets`（含元数据）、`integration`、`governance`、`services`、`admin`。

- **`governance` 最重**，同时承载**定义域**（标准 / 参考数据 / **主数据**）与**管控域**（质量 / 血缘 / 安全 / 策略 / 生命周期）。域内用**能力键**区分粒度（两级：域 + 键），如 `arda.governance.master_data` / `.standards` / `.quality` / `.lineage` / `.security` / `.policies` 各自可单独开通 / 定价，**无需为主数据/标准拆独立域**。
- **主数据归 `governance`，不归 `assets`**（理由见 `biz-230` §1）：资产域=有哪些数据；治理域=哪份权威 / 可信。
- **可选拆分**：若产品把「主数据 / 数据标准」当旗舰模块单独售卖，可拆出**第 6 域「数据定义（标准+主数据）」**，代价是门控矩阵增一列。**建议先用能力键分层，确有需要再拆域**（先键后域，避免过早增加门控复杂度）。

### 1.2 两轴模型：能力域（纵）× 应用视角/生命周期（横）

**两轴是「能力域（纵向）× 应用视角（横向）」的矩阵**，不是把能力压成一个被动维度、只按域列清单（前一版把纵轴丢了，此处纠正）。

- **纵轴 · 能力域**：一个数据能力（接入 / 资产·元数据 / 标准 / 主数据 / 质量 / 血缘 / 安全 / 生命周期）**贯穿它自己的完整生命周期**：**目标 → 过程 → 结果 → 服务 → 监管**。固定一个能力**纵向看** = 该能力的**端到端专题**（适合能力负责人：质量经理、安全官、主数据 steward）。
- **横轴 · 应用视角（生命周期阶段）**：固定一个阶段**横向拉通所有能力** —— 这就是**功能域**：
  - **目标**（定义 / 规划）：定标准 / SLA / 分级策略 / 主数据模型 / 接入计划。
  - **过程**（执行）：落标 / 跑质检 / 脱敏加密 / 采血缘 / 匹配合并 / 登记同步（治理域工作台 + 集成执行）。
  - **结果**（看）：标准符合性 / 质量分 / 分级状态 / 血缘图 / 金记录 / 资产画像（资产域）。
  - **服务**（用）：可信数据发布为 API / 数据产品 / 主数据服务（服务域）。
  - **监管**（审计 / 配置）：变更 / 访问审计、留存销毁、配置合规（管理域）。

**能力 × 阶段 矩阵**（每格 = 该能力在该阶段的形态；**横读=某阶段拉通所有能力=功能域，纵读=某能力端到端=能力专题**）：

| 能力域＼阶段 | 目标（定义） | 过程（执行） | 结果（看） | 服务（用） | 监管（审计/配置） |
|---|---|---|---|---|---|
| 数据接入 | 接入规划 / 源清单 | 登记 / 连接 / 同步 | 入湖 / 自动编目 | 供下游源 | 连接 / 接入审计 |
| 资产 / 元数据 | 编目体系 / 分类 | 编目 / 打标 / 术语 | 资产画像 / 目录 | 目录发现 | 资产变更审计 |
| 数据标准 | 定标准 / 数据元 | 落标 / 评审 | 标准符合性 | 标准约束 | 标准合规审计 |
| 主数据 | 主数据模型 / 权威源 | 匹配 / 合并 / steward | 金记录 | 主数据服务 | 主数据变更审计 |
| 数据质量 | 质量目标 / SLA | 建规则 / 跑质检 | 质量分 / 报告 | 质量准入 | 质量合规审计 |
| 数据血缘 | 血缘覆盖目标 | 采集 / 校正 / 影响 | 血缘图 | 可溯源服务 | 血缘审计 |
| 数据安全 | 分级 / 合规目标 | 脱敏 / 加密 / 授权 | 分级 / 敏感状态 | 安全过滤服务 | 访问审计 / 安全配置 |
| 生命周期 | 留存 / 销毁策略 | 归档 / 销毁执行 | 留存状态 | 可用性影响 | 生命周期审计 |

**功能域（横）= 生命周期阶段** 的具体命名对齐（承 §1.1 的 5 域）：

| 阶段（横） | 功能域 | 说明 |
|---|---|---|
| 结果（看） | **资产域** | 所有能力的结果汇成数据画像 |
| 过程（执行）+ 目标（定义） | **治理域** | 定目标 + 执行治理的工作台 |
| （接入能力的执行入口） | **集成域** | 数据接入这一能力的过程面独立成域 |
| 服务（用） | **服务域** | 所有能力的服务化出口 |
| 监管（审计/配置） | **管理域** | 所有能力的审计 / 配置 / 运营 |

**产品要同时支持两种导航**：
- **横向（按阶段 / 域）**：一个角色在一个域内拉通所有能力（消费者在**资产域**看所有数据的所有结果面；steward 在**治理域**做所有能力的治理）。→ 主 IA（§2 各板块）。
- **纵向（按能力）**：一个能力负责人端到端管一个能力（质量经理看质量的 目标→过程→结果→服务→监管 全链）。→ **能力专题视图**（二级导航 / 过滤，跨域拉一条能力线）。

> **门控不变**：能力键仍门控"能力"；同一能力在不同阶段的门控轻重不同——**结果（看）轻**（资产 baseline），**过程（管）重**（该能力键 + 更高角色），**监管**属管理域（权限为主）。与 functional-domains §3-4 一致。
> **数据目录 vs 服务目录**：资产域=**数据目录**（`Dataset`，看数据）；服务域=**服务/API 目录**（`DataService`，取数据）——勿混。
> 采纳影响：`biz-210`（资产域=结果）纳入各能力的**只读结果面**；`biz-230`（治理域=过程+目标）保留**执行/定义面**；并新增**能力专题（纵向）**作为二级导航。板块重排见 `biz-110` §5 与 `biz-300`。

## 2. 价值链：编目 → 治理 → 服务

arda 的能力不是并列的五块，而是一条**递进的数据价值链**：

```
① 集成/资产: 把外部数据登记、编目为 Dataset(有口径/负责人/分级)
        ↓
② 治理:      给资产加标准/质量/血缘/策略 → 数据"可信"
        ↓
③ 服务:      把可信资产封装为 API/查询/导出 → 数据"可用"
        ↓
④ 消费:      人在控制台用, 或外部(含智能体)经服务契约取用(可溯源)
```

- **越往右越有商业价值**：裸数据源（左）→ 可信数据服务（右）。定价与档位通常沿此链递增（见 `ent` 维度）。
- **治理是链条枢纽**：没有 ② 的资产只是"堆着的数据"；②让 arda 区别于普通数据目录——**治理即信任**（见 §3.3）。

## 3. 跨切面（所有板块共用的横向机制）

### 3.1 两级隔离（org 硬 / workspace 默认软）

**org（tenant）= 硬隔离**：任何数据访问绝不跨 org。**workspace = 默认软隔离**：隔离键 = `workspaceId`（=平台/IdP `active_workspace`），所有板块的每个业务实体带 `workspaceId`、查询默认强制按其过滤、workspace 内唯一；同 org 内可经显式授权跨 workspace 访问（资源级 `WorkspaceGrant`，权益不随授权流动，见 [`data-160`](arda-data-160-cross-workspace-authorization.md)）。机制见 [`data-110`](arda-data-110-isolation.md)。**本层不重述，各板块详细设计默认遵守。**

### 3.2 两轴门控（订阅 × 权限）

任何能力的"可用"= **订阅维度**（买了没：平台下发的 `(workspace,product=arda)` 商业事实 status×tier，经 arda 本地能力矩阵展开为 feature-key，见 [`ent-110`](arda-ent-110-local-implementation.md) §2a，2026-07-13 修订）**AND** **权限维度**（角色够不够：`session.roles`）。三层防御：导航可见性 / 路由布局校验 / 动作与配额校验（配额数值由平台下发：上限 `limits` 本地执行、消耗池经 consume 放行）。机制见 [`arda-functional-domains-and-entitlement.md`](arda-functional-domains-and-entitlement.md) §3-4。**各板块详细设计只声明"用哪些键 + 哪些动作需要更高角色"，不重述机制。**

### 3.3 治理即信任层

分级(`classification`)/策略(`Policy`)/质量(`QualityResult`)/血缘(`LineageEdge`)是数据"可信可溯源"的控制面：对内标注资产可信度，对外（被消费/被智能体取用时）按分级脱敏、随数据传递来源。这是 arda 的差异化，详见 [`arda-data-platform-agent-support.md`](arda-data-platform-agent-support.md) §2。

### 3.4 对外数据契约

数据经 `DataService`(rest_api/query/export/share) + `ApiKey` 对外；对外时门控不变量（隔离/权益/分级过滤/审计/配额）必须在 arda 侧收口。详见 [`arda-data-platform-agent-support.md`](arda-data-platform-agent-support.md) §3。

### 3.5 可推导优于可存储

能从其他表算出的值（质量总分、治理覆盖率、调用量、订阅数）不落库，UI 展示时聚合。避免配置漂移。原则见 [`data-100`](arda-data-100-architecture.md) §1。

## 4. 目标态架构（能力 → 屏幕 → 数据 → 门控）

```
用户/消费方
   │
   ▼ 会话(AccountGate) + 整站权益(EntitlementGate) + 板块级两轴门控(§3.2)
┌─────────────────────────────────────────────────────────────┐
│ 板块屏幕层  catalog · (etl/sources) · standards/quality/       │
│            lineage/security · service · (admin: keys/audit)    │
├─────────────────────────────────────────────────────────────┤
│ 服务端数据层 (app)/<板块>/data.ts —— 强制 where{workspaceId}    │
├─────────────────────────────────────────────────────────────┤
│ 领域模型 (Prisma/Postgres, arch 维度)                            │
│   Dataset/Tag/Glossary · DataSource · Policy/Quality/Standard/ │
│   Lineage · DataService · ApiKey/AuditLog · WorkspaceRef/Seed  │
└─────────────────────────────────────────────────────────────┘
   │ 复用平台服务
   ▼ 身份/隔离(IdP) · 权益/计量(平台, ADR-11) · Redis(会话) · 存储
```

板块内部（屏幕/数据/交互/门控/待办）下钻见第 2 层各文档。

## 6. 与既有架构的关系

- **da（数据架构）已完成**：本业务架构的数据模型**以 arch 维度为准**，不重定义 schema；本层只谈"能力/板块/价值链/门控落点"。
- **en（权益）复用**：门控机制不在本系列发明，引用 `ADR-11` + functional-domains。
- **本系列新增的是"业务视角"**：把已有实体/屏幕组织成能力全景与价值链，指导后续板块详细设计与落地。
